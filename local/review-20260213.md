# PreVABS Code Review — 2026-02-13

## Overview

This review covers the full PreVABS codebase: ~40 source files, ~25 integration test cases, CMake build system, and dependencies. The project is a mature-enough engineering tool but carries significant accumulated technical debt — primarily in memory management, null-safety of XML parsing, and a large volume of commented-out dead code from an in-progress API migration.

---

## Critical Issues (Fix Immediately)

### 1. Memory Leaks — `main.cpp:282,289`
`Message*` and `PModel*` are heap-allocated but never deleted:
```cpp
Message *pmessage = new Message(config.file_name_log);  // never freed
PModel *pmodel = new PModel(config.file_base_name);     // never freed
```
**Fix**: Use `std::unique_ptr<Message>` and `std::unique_ptr<PModel>`.

### 2. Null Pointer Crashes on Malformed XML
`PModelIOReadCrossSection.cpp` never checks whether `first_attribute()` or `first_node()` returns `nullptr` before dereferencing:
- Line 98: `p_xn_sg->first_attribute("name")->value()` — crashes if `name` attribute is absent.
- Line 321: `nodeGeneral` is used without a null check; if `<general>` is missing, all subsequent reads crash.
- Line 558: `nodeLEWeb->first_node("pos_top")->value()` — unchecked inner node.

**Fix**: Add null checks and throw a descriptive `std::runtime_error` on missing required XML nodes/attributes.

### 3. `argv` Out-of-Bounds Access — `main.cpp:103`
The legacy `parseArguments()` function reads `argv[i+1]` without checking that `i+1 < argc`. Passing `-i` as the last argument triggers undefined behavior.

**Fix**: Complete the migration to CLI11 (already vendored and partially integrated) and remove `parseArguments()`.

### 4. Shell Injection — `execu.cpp:95`
```cpp
std::system(command.c_str());
```
Commands are built from user-supplied file paths with no sanitization. A filename containing `;`, `&`, or `|` would execute arbitrary shell commands.

**Fix**: On Windows use `CreateProcess`; on POSIX use `fork`/`execvp` with an argument array.

### 5. `PDCELHalfEdgeLoop` Objects Leaked in `~PDCEL`
`PDCEL::~PDCEL()` deletes vertices, half-edges, and faces, but `_halfedge_loops` — a `std::vector<PDCELHalfEdgeLoop*>` — is not iterated and deleted.

**Fix**: Add `for (auto *loop : _halfedge_loops) delete loop;` in `~PDCEL`, or use `std::vector<std::unique_ptr<PDCELHalfEdgeLoop>>`.

## High Severity Issues

### 6. `declarations.hpp` Missing `#pragma once`
Every other header uses `#pragma once`. `declarations.hpp` has no include guard and `#include`s nearly the entire header tree. Multiple inclusion causes redefinition errors in any TU that inadvertently includes it twice.

**Fix**: Add `#pragma once` as the first line.

### 7. `log_severity_level` Declared But Never Defined
`globalVariables.hpp:66` has:
```cpp
extern std::string log_severity_level;
```
No translation unit defines this variable. Any code that takes its address or uses it as an lvalue will cause a linker error.

**Fix**: Either define it in `globalVariables.cpp` (or `utilities.cpp`) or remove the declaration.

### 8. `atoi()` Used to Parse `double` Threshold Values — `PModelIOReadCrossSection.cpp:441,452`
Variables declared as `double` are parsed with `atoi()`, silently truncating the fractional part. Over 100 additional `atof()`/`atoi()` calls across IO files provide no error feedback on bad input.

**Fix**: Replace with `std::stod()`/`std::stoi()` inside try/catch blocks and report parse failures with the offending XML element path.

### 9. `fopen()` Return Value Unchecked — `PModel.cpp:43`
If `fopen` fails (e.g., read-only directory), `config.fdeb` is `nullptr`; subsequent `fprintf(config.fdeb, ...)` calls are undefined behavior (crash).

**Fix**: Check the return value and either throw or disable debug output if the file cannot be opened.

### 10. `_dupenv_s` Buffer Leaked on Windows — `execu.cpp:63`
`_dupenv_s` allocates a buffer that the caller must `free()`. The buffer is passed to `_putenv_s` but never freed.

**Fix**: `free(path)` after `_putenv_s(...)`, or use `std::string` + `getenv()`.

### 11. `CrossSection` and `LayerType` Objects Leaked Across Calls
`readCrossSection()` creates a new `CrossSection*` each call and hands it to `PModel`, which does not delete the previous one. `PModel` has no destructor for `_layertypes`. Both `homogenize()` and `dehomogenize()` call `readInputMain()`, doubling the leak.

## Medium Severity Issues

### 12. PI Constant Insufficient Precision — `globalConstants.hpp:21`
```cpp
const double PI{3.14159265};
```
Only 9 significant figures vs. double's 15–16. Use `M_PI` (POSIX) or `std::numbers::pi` (C++20) or at minimum `3.141592653589793`.

Additionally, `PModelIOReadCrossSection.cpp:563` re-defines PI locally as `atan(1) * 4`, shadowing the global and creating inconsistency.

### 13. `PConfig.hpp` Is a Dead File
`include/PConfig.hpp` contains an empty class `PConfig {}`. The actual `PConfig` struct is defined in `globalVariables.hpp`. The empty file is misleading.

**Fix**: Delete `PConfig.hpp` (or merge the real struct into it and include it where needed).

### 14. Massive Commented-Out Dead Code
Hundreds to thousands of lines of commented-out code exist across source files:
- `PModel.cpp:383–757`: ~375 lines of old GModel/GEdge Gmsh API code.
- `execu.cpp:117–447`: ~330 lines of Fortran integration and old ShellExecute.
- `PModelIO.cpp:79–297`: old SG reading code.
- `src/vabs/output.f90`: a Fortran file that is never compiled.

This obscures intent and raises maintenance cost. **Fix**: Remove all commented-out code; git history preserves it.

### 15. CMake `GLOB_RECURSE` for Source Files
`file(GLOB_RECURSE SOURCES ...)` is explicitly discouraged in CMake documentation. Adding or removing a `.cpp` file does not trigger a CMake re-run, leading to stale builds.

**Fix**: List source files explicitly in `CMakeLists.txt`.

### 16. No Compiler Warning Flags
No `-Wall -Wextra` (GCC/Clang) or `/W4` (MSVC) flags are set. Many of the issues above would be caught at compile time with warnings enabled.

**Fix**: Add `target_compile_options` with appropriate warning flags per platform.

### 17. Old CMake Minimum Version
`cmake_minimum_required(VERSION 3.0)` — CMake 3.0 was released in 2014. The build already uses features requiring 3.14+.

**Fix**: Bump to `cmake_minimum_required(VERSION 3.14)`.

### 18. Circular Include Graph via `declarations.hpp`
`declarations.hpp` includes almost every header; each of those headers includes `declarations.hpp`. This deep circular include graph inflates compilation times and hides true dependency relationships.

**Fix**: Refactor to include only what each file directly needs.

## Low Severity / Style Issues

### 19. Mixed Naming Conventions
The style guide mandates `_` prefix for member variables and PascalCase for functions. Several classes deviate:
- `Segment`: members `slevel`, `slayupside` lack the `_` prefix.
- `Filling`: members `fname`, `p_fbaseline` use inconsistent Hungarian-style.
- Getter naming: `Material::getName()` vs `PDCELVertex::name()` — mix of `getX()` and `x()` styles.

### 20. `using namespace rapidxml` in Headers
`utilities.hpp:20` and `PModelIO.hpp:28` have `using namespace rapidxml` at file scope in a header. This pollutes every TU that includes them.

**Fix**: Use the full `rapidxml::` prefix in headers.

### 21. `PModel` God Object
`PModel` has 60+ public methods spanning IO, geometry building, Gmsh interaction, execution, and post-processing. Splitting into `PModelBuilder`, `PModelIO`, and `PModelRunner` collaborating objects would improve testability and maintainability.

### 22. Dual CLI Parser System
`main.cpp` has both the new CLI11-based parser (commented out) and the legacy `parseArguments()`. The dual system creates confusion.

**Fix**: Activate CLI11, delete `parseArguments()`.

### 23. Non-Reentrant Static Class Counters
`CrossSection::used_material_index` and `PComponent::count_tmp` are global static counters. Processing multiple cross-sections in one session without resetting state will produce incorrect output.
